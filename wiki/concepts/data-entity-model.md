# Data Entity Model

## Definition

The canonical set of data entities that must exist in the system's data layer to support the full Multi-Tenant Adaptive Lead Intelligence Engine. Organized in three groups: core lead lifecycle, governance & quality, and operational observability & delivery. Every claim, score, feedback signal, and operational event has a named home in this model.

## Group A — Core Lead Lifecycle

These 15 entities cover everything from a lead entering the system through to a scored, bucketed, and consent-checked output.

| Entity | What it is | Why it is needed |
|---|---|---|
| `tenant` | The business account / client workspace | Every piece of data must belong to the correct business boundary |
| `business_profile` | Structured profile of the tenant's own business (from web research + manual input) | System must understand the tenant's business before it can infer their ideal customer |
| `business_profile_source` | Evidence sources used to build the business profile (website, LinkedIn, Wikipedia, manual answers) | Provenance, trust, and future update tracking |
| `ideal_customer_profile` | Inferred target-customer definition derived from the business profile | Lead evaluation is based on who is most likely valuable for that specific business |
| `lead` | The canonical prospect record | Everything revolves around the lead — scoring, enrichment, workflow |
| `contact_identity` | Normalized phone, email, social identifiers | Deduplicates the same person across multiple channels; stable identity anchor |
| `lead_source` | Channel / mechanism through which the lead entered the system | Attribution and source-quality analysis |
| `event` | Time-based action or interaction (message received, form submission, referral, abandoned cart) | Behaviour, engagement, and intent are visible through the lead's timeline |
| `lead_enrichment` | Expanded profile built from multiple internal and external data points | Raw lead data is too thin for scoring; enrichment provides the facts the scoring prompt needs |
| `enrichment_source` | Provenance of each enriched field (where each fact came from) | Explains and audits the enriched data used for scoring |
| `signal` | Evaluation questions generated by the Signal Agent from the ICP, organized by scoring dimension | Defines the evaluation framework — what questions to ask about each lead |
| `signal_evaluation` | Per-lead answer or result for each signal question after checking enriched data | Storing how each lead performed on each question; the filled-in blanks |
| `lead_score` | Final scoring outcome: dimension scores, overall score, bucket | The business needs a final prioritization result to drive follow-up |
| `score_model_version` | Version of the scoring logic / thresholds / model used to produce a score | Scores remain reproducible and auditable even when the scoring method changes |
| `consent_preference` | Lead's communication permissions and preferred contact channels | Even a HOT lead should only be contacted through allowed and preferred channels |

## Group B — Governance & Quality

These 6 entities back the governance layer's six domains (monitoring → auditability → lineage → feedback loops → quality tracking → security). Full governance documentation: [[analyses/governance-observability-layer]].

| Entity | What it is | Why it is needed |
|---|---|---|
| `audit_log` | Chronological record of important system actions (create, update, delete, access, export, retry, policy change) with actor, target, timestamp | Answers "who did what, when, to which record" — essential for accountability and enterprise auditability |
| `tenant_policy` | Stored rule set defining how a tenant's data and workflows should be handled (retention, masking, allowed integrations, review requirements) | Different tenants may require different operating rules; rules must be explicit, versioned, and queryable |
| `quality_rule` | Formal definition of what good data or acceptable output means (required fields, freshness limits, allowed values, duplication thresholds) | Quality checks require clear standards; otherwise they remain scattered in application logic and unmanageable |
| `quality_check_result` | Stored result of running a quality rule on actual data or pipeline outputs (status, measured value, timestamp, issue count) | Defining a rule is not enough; the system must record whether it passed or failed over time |
| `feedback_record` | Structured record of feedback on a score or recommendation (useful, wrong, corrected, approved) | Feedback should be usable system data, not loose comments — enabling scoring quality improvement over time |
| `lineage_record` | Trace record linking source object/field to target object/field, capturing how data moved through enrichment, scoring, aggregation, or reporting | Answers "where did this output come from and how was it produced" — critical for debugging, compliance, and explainability |

## Group C — Operational Observability & Delivery

These 11 entities back runtime visibility, reliability measurement, and output delivery.

| Entity | What it is | Why it is needed |
|---|---|---|
| `business_profile_version` | Versioned snapshot of the tenant's business profile at a point in time | Business persona changes affect ICP, signals, and scoring behaviour; must preserve which understanding was active when a score was produced |
| `ideal_customer_profile_version` | Versioned snapshot of the tenant's ICP at a point in time | ICP changes directly alter how leads are judged; must preserve the exact ICP version that influenced a signal set and score |
| `pipeline_run` | Top-level record of one complete workflow execution (ingestion, enrichment, scoring, report generation) | One clear place to see whether a full business workflow actually ran successfully for a tenant |
| `task_execution` | Record of one step inside a pipeline run (validate, enrich, score, report, sync) with step-level status, retries, errors, duration | Most failures happen at the step level; identifies exactly where a workflow failed or slowed down |
| `telemetry_event` | Structured operational event emitted by the system during runtime (severity, service, event type, tenant context, execution context) | Detailed runtime visibility for troubleshooting; what exactly happened inside a workflow while it was running |
| `alert_incident` | Tracked issue created when a failure, threshold breach, or service problem occurs (repeated scoring failure, missed report time) | Converts important technical failures into actionable, trackable issues; not every log should be treated equally |
| `slo_measurement` | Record showing whether a service-level target was met during a given time window (latency, completion rate, delivery time) | Measures reliability in a meaningful way — not just raw failures but whether expected service quality was delivered |
| `dashboard_definition` | Configured dashboard view defining what a tenant or internal user sees (lead summaries, scores, filters, KPI sections) | Dashboards are reusable delivery artifacts, not just UI screens; the platform needs a defined way to present processed outputs |
| `notification_delivery` | Record of a notification the system sent or tried to send (channel, recipient, status, time) | Track whether messages were actually sent, failed, or need retry |
| `delivery_endpoint` | Configured external destination where the system sends outputs (webhook URL, partner endpoint, external system target) | External outputs must go to known, managed destinations rather than hardcoded targets |
| `report_definition` | Configured report defining what is generated, how often, what time window, what content, what format | Reports are formal delivery artifacts, not ad hoc queries; enables periodic client-facing lead outputs |

## Cross-Layer Connections

| Intelligence Layer concept | Data entity |
|---|---|
| ScoringOutput | `lead_score` |
| schema_version + prompt_version fields | `score_model_version` |
| PersonaObject (persona engine) | `business_profile` + `ideal_customer_profile` |
| EnrichedLead (passed to score_lead) | `lead_enrichment` |
| Signal questions (fill-in-the-blanks prompt) | `signal` |
| Signal values (filled in at enrichment) | `signal_evaluation` |

| Pipeline / governance concept | Data entity |
|---|---|
| Pipeline 1 / Pipeline 2 run record | `pipeline_run` |
| Per-stage execution record | `task_execution` |
| Lineage written at every stage | `lineage_record` |
| Governance feedback loop raw input | `feedback_record` |
| Quality metrics computation | `quality_rule` + `quality_check_result` |
| Monitoring and alerting | `telemetry_event` + `alert_incident` |
| SLO tracking | `slo_measurement` |

## Why It Matters

The entity catalog is the contract between all three layers. Without named entities with explicit "why is it needed" rationale:
- Storage decisions drift (data that should be in `signal_evaluation` ends up only in `pipeline_log` prompts)
- Governance checks can't be automated (quality rules have no table to run against)
- Historical explainability breaks (scores can't be replayed without `score_model_version` and `business_profile_version`)

## Open Questions

- Schema for `signal` — does it include the dimension type (fit/intent/etc.) and the detection_rule format? Detection rule format is a hard blocker for Lead Enrichment. See [[analyses/orchestration-layer-dependencies]].
- Does `quality_rule` store rules as SQL, as threshold + field references, or as free text?
- How does `report_definition` relate to `dashboard_definition` — subset or separate artifact?
- What triggers `alert_incident` creation — governance monitoring job, or `task_execution` failure directly?
